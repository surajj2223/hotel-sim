Real HTTP integration — by design

payments-sim is a separate Spring Boot service. core-api integrates with it over real HTTP + webhooks. This is deliberate — to simulate a real PSP integration, not a mock. The same discipline would apply to a real Adyen integration.

Reference taxonomy

Payment-link flow

1

Caller creates payment. ops-web or MCP calls POST /bookings/{id}/payments with shopperReference, merchantReference, amount, captureMode.

2

core-api creates Payment record (status: PENDING_LINK) and calls POST /payment-links on payments-sim.

3

payments-sim mints paymentLinkId and returns { paymentLinkId, checkoutUrl }. core-api stores paymentLinkId on the Payment and returns the checkout URL to the caller.

4

Customer visits pay-web (hosted checkout) and completes payment.

5

payments-sim fires webhook to core-api /webhooks/payment. For IMMEDIATE: fires AUTHORISATION then CAPTURE. For MANUAL: fires AUTHORISATION only; CAPTURE follows when operator explicitly captures.

6

core-api handles webhook idempotently: matches by merchantReference, stamps pspReference, updates Payment status, posts to ledger via outbox.

Webhook event vocabulary

Event codeTriggered whencore-api response
AUTHORISATION PSP authorises the payment Stamp pspReference; status → AUTHORISED
AUTHORISATION_FAILED Authorisation declined Status → FAILED
CAPTURE Payment captured Post revenue to ledger; status → CAPTURED
CAPTURE_FAILED Capture attempt failed Status → CAPTURE_FAILED
CANCELLATION Auth cancelled before capture Status → CANCELLED; no ledger reversal
REFUND Refund settled by PSP Post reversal to ledger; Refund → REFUNDED
REFUND_FAILED Refund attempt failed Refund → REFUND_FAILED
OFFER_CLOSED Auth expired with no capture Status → EXPIRED; no ledger reversal

All webhook payloads echo merchantReference, pspReference, and shopperReference.

Idempotency rules

1

Webhooks carry a pspReference-derived idempotency key.

2

core-api matches inbound webhook to a Payment by merchantReference.

3

If a webhook with the same pspReference and event code has already been processed, the handler returns 200 OK with no state change.

4

Duplicate processing is a no-op — never double-posts to the ledger.
