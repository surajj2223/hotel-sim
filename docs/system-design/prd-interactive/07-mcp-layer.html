Last, lightest piece

The MCP server is Head 2 — a thin tool-wrapper over the same core-api endpoints that ops-web already uses. It adds no new logic, no new endpoints, no new persistence. It is the last, lightest piece of the system — built only after the full smoke test is green.

Additive-only constraint

No capability may exist solely for the agent. Every MCP tool maps 1:1 to an existing core-api endpoint that already fully serves ops-web. The MCP server calls core-api over the same HTTP calls ops-web would make.

This PRD is Wave 0 forward-documentation

This PRD is written now (Wave 0) to document the design intent and the three proof capabilities. Implementation is Wave 2. Stack and port are TBD.

The three capabilities that prove the thesis

1. Preference-aware suggestion

The API returns availability alongside stored customer preferences. The LLM does the matching and judgement — not the server. Facts from the system, judgement from the model.

"Mr. John Patel checks in Friday for 3 nights, wants a high-floor quiet room; add a couples massage Saturday afternoon and a table for two at 8."

Agent calls: searchAvailability → getCustomerPreferences → proposes → createBooking (on confirm).

2. Cross-vertical orchestration

One natural-language instruction produces a room + spa + dinner folio in a single confirmed write. Demonstrates that the unified Booking/folio model is not just a design principle but an operational reality.

Multi-vertical folio created atomically via one POST /bookings, not as three separate bookings.

3. Financial read-back

Conversational interrogation of the ledger. Read-only — no human gate required.

"Pull up everything for Mrs. Chen this stay and tell me what's unpaid."
"Today's revenue split by vertical, and flag anything still pending payment."

Calls: getFolio, getRevenue, listUnpaidBookings.
