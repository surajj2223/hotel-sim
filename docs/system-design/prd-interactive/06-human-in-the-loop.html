Writes are never autonomous

The MCP performs writes, but a human confirms before anything with repercussions. Safety lives in the server — not in the MCP behaving well. An agent that misbehaves cannot write without the human-authorisation signal.

The four mechanisms

#MechanismWhere it livesWhat it prevents
1 Confirm-then-commit UX flow / agent conversation Autonomous write without human awareness
2 Write-time revalidation core-api — every write handler Writing stale data after a race condition
3 Server-side commit gate core-api — application service layer Agent self-authorising a repercussive write
4 Dual-channel confirmation Both ops-web and MCP conversation MCP having privileged write paths not available to ops-web

Safety invariants (non-negotiable)

1

No write may succeed without passing write-time revalidation.

2

No repercussive write may succeed without a human-authorisation signal.

3

The agent cannot self-mint the human-authorisation signal.

4

ops-web click and agent "yes" produce identical ledger outcomes.

5

A failed write must return the current state — enabling informed retry.

Explicitly deferred

Soft holds / drafts (TTL'd inventory reservation) Multi-approver workflows Audit log beyond outbox/event log

The availability seam is kept clean so holds are purely additive later — a hold becomes "another thing that consumes availability."

Mechanism 1: Confirm-then-commit

No draft ceremony

No two-phase writes. No provisional state that must be cleaned up on timeout. The agent reads availability, proposes in chat, the human says "yes", then the agent calls one write endpoint.

Flow

1

Agent calls searchAvailability (read — no gate).

2

Agent proposes options in chat. Human reviews.

3

Human says "yes" (the authorisation signal).

4

Agent calls POST /bookings (single call, with human-auth signal).

5

core-api revalidates atomically → writes or returns 409 with current state.

If the write returns 409, the agent reports the changed state to the human and may re-propose. No silent retry.

Mechanism 2: Write-time revalidation

The core safety mechanism

Every write atomically re-checks availability and price. If state has moved since the caller last read, the write returns 409 Conflict with the current state — never silently writing stale data.

Why this design?

Cheaper and more reliable than distributed locks or draft reservations. The availability window between read and confirm is acceptable because this check catches any movement. The caller receives actionable information to retry intelligently.

Applies to

All writes in the capability surface (HS-05). No write bypasses revalidation.

On conflict

Returns 409 Conflict with the current state of the relevant resource (current availability, current prices). This enables an informed retry or updated proposal to the human.

Mechanism 3: Server-side commit gate

Safety lives in the server

core-api requires a human-authorisation signal on repercussive writes. The agent cannot self-mint this signal. An agent that hallucinated a "yes" or looped cannot write.

Signal sources (both are legitimate)

SourceHow the signal is produced
ops-web A UI button click. The front-end generates or includes the authorisation signal in the request.
MCP agent conversation The human's "yes" in the conversation. The agent forwards it; it cannot fabricate it.

The specific mechanism (header, field, token) is defined in contracts/WAVE0_02_OPENAPI.yaml.

Mechanism 4: Dual-channel confirmation

Part of the proof

Demonstrating that the same createBooking called from a button and from the agent lands identically in the ledger is itself a slice of the proof. The two channels converge at the same endpoint with the same rules applied.

The equivalence

ChannelHuman actionResult in ledger
ops-web Click "Confirm Booking" button Booking created, inventory consumed, folio updated
MCP agent Type "yes" in conversation Identical: Booking created, inventory consumed, folio updated

No capability may exist solely for the agent. No write path for the agent that is not also available to ops-web.

Standing rule

Frozen contracts may not drift silently. Any deviation a builder must make is recorded here with date, the clause affected, what changed, and why. Until a row is arbitrated and the contract re-frozen, the code is the source of truth for that clause.
