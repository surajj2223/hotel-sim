The Thesis

Here is a normal, complete, headful operations system — the kind that already exists everywhere — and an MCP layer drops onto it cleanly to make it headless, without rebuilding anything underneath.

Persuasive power comes from the UI being real and unremarkable. The MCP is strictly additive: it wraps the same endpoints ops-web already uses. No capability may exist solely for the agent.

What this system is

A proof-of-concept multi-vertical hotel operations platform. It manages bookings, inventory, payments, and financial ledger postings across four sellable verticals from a single core-api body — served by both a full operations console (ops-web) and, in Wave 2, a headless MCP agent.

The four verticals

VerticalWhat is soldDefault capture
Rooms Room / night booking MANUAL — auth now, capture at checkout
Spa Therapist and treatment time slots TBD by strategy
F&B Restaurant covers per service period IMMEDIATE — auth and capture together
Events Capacity-limited experiences (city tours, horse rides) TBD by strategy

Every vertical is the same thing at the domain level: a bookable resource with capacity over time, sold to a customer, paid for, and posted to a ledger. Vertical-specific behaviour lives in per-vertical strategy classes — not in parallel silos.

Two equal heads, one body

LayerProjectDescription
Body core-api All logic, rules, and state. The single source of truth. Spring Boot 3.5, Java 17.
Head 1 (headful) ops-web A complete standalone operations console. React. Every capability has a screen.
Head 2 (headless) MCP server A thin tool-wrapper over the same core-api endpoints. Wave 2. Strictly additive.

The symmetry rule

Acceptance test for every endpoint: can ops-web build a complete screen on it? If yes, the MCP will be fine. No capability may exist solely for the agent.

Core entities

Customer
Identity + contact. Carries an immutable, opaque shopperReference (SHPR-…), one-to-one with the customer, never the raw DB id. Reconciliation never uses raw id.
CustomerPreference
Flexible key/value store, cross-vertical (e.g. floor=high, dietary=vegan, spa_therapist=female).
Product (polymorphic)
Keyed by vertical, with vertical-specific config. A Deluxe Room, a 60-min massage, a dinner slot, and a horse ride are all Products — same abstraction, different strategy.
Booking / Folio
One customer + one or more BookingLines, each referencing a Product, time window, quantity, and a price snapshot. One booking can span verticals (room + spa + dinner = one folio). Carries totalAmount, amountPaid, amountRefunded, and derived balance. Paid = balance == 0, not a boolean.
Inventory / Availability
Generic capacity per product/slot. Consumed by committed bookings only. No holds in this POC. The availability seam is kept clean so holds are purely additive later.

Per-vertical strategy interface

Each vertical implements a common strategy interface supplying:

  • Availability calculation
  • Pricing
  • defaultCaptureMode() — the vertical's payment capture default

This is where "Rooms behave this way, F&B behaves that way" lives — in one place, not in parallel silos. See HS-02 for detail.

Applied patterns

Hexagonal layering Strategy per vertical DTOs at every boundary Idempotent webhook handling Outbox / event log Write-time revalidation Server-side commit gate

Deliberately out of scope (POC)

Do not build these

  • Microservice-per-vertical
  • Event bus / Kafka
  • CQRS
  • Real authentication (stubbed only)
  • Real money / multi-currency
  • Holds / drafts (inventory reservation with TTL)
  • Multi-capture / incremental auth
  • Scaling concerns

The availability seam is designed so holds are purely additive later — a hold becomes "another thing that consumes availability." It is not missing; it is deferred by design.

Settled decisions (quick reference)

#Decision
1 4 verticals behind a unified Booking/folio; polymorphic Product; per-vertical strategies.
2 shopperReference is immutable and opaque. Reconciliation anchors on merchantReference. PSP mints pspReference and paymentLinkId.
3 One booking → many payments; partial payments supported; refunds are children of a payment via originalReference.
4 Partial capture in; multi-capture and incremental-auth out.
5 Ledger posts on capture, not auth.
6 ops-web is a complete product; MCP is strictly additive; no agent-only endpoints.
7 Stack: core-api + payments-sim (Spring Boot), ops-web + pay-web (React), Postgres on Docker, docker-compose.

Standing rule

Frozen contracts may not drift silently. Any deviation a builder must make is recorded here with date, the clause affected, what changed, and why. Until a row is arbitrated and the contract re-frozen, the code is the source of truth for that clause. This is the hotel-sim analogue of NikkFit's "Code-vs-PRD Errata," but it governs a frozen contract, so it doubles as the central-arbitration record.
