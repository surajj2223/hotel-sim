First-class feature

Finance is a first-class showcase feature, not credibility dressing. The ledger is fully traceable from booking through to PSP transaction, reportable by vertical, and correctly handles partial capture, partial refunds, and multi-payment folios.

Payment reference taxonomy

ReferenceMinted byMeaningMutability
shopperReference core-api Stable customer identifier sent to PSP; on Customer entity. Immutable — never changes.
merchantReference core-api Our reference per payment attempt; reconciliation anchor. Immutable per attempt.
pspReference payments-sim PSP's own transaction id; returned on first webhook. Minted by PSP; we store and never alter.
paymentLinkId payments-sim PSP's id for a hosted payment link. Minted by PSP; we store and never alter.

Reconciliation key

Reconciliation anchors on merchantReference. Customer continuity across stays anchors on shopperReference.

Amount tracking on Booking

FieldFormula
totalAmountSum of all BookingLine price snapshots
amountPaidSum of all captures across all Payments for this Booking
amountRefundedSum of all Refunds across all Payments
balancetotalAmount − amountPaid + amountRefunded

Paid = balance == 0

One Booking may have many Payments. A Payment settles exactly one Booking. No many-to-many.

Capture modes

ModeBehaviourRevenue postingDefault vertical
IMMEDIATE Auth and capture happen together. Posts at auth+capture moment. F&B
MANUAL Auth now (card hold), capture later via explicit operator action. Posts only at capture time. Rooms

Default is a default

The capture mode default is supplied by the vertical strategy (defaultCaptureMode()), but it is explicit and overridable on the payment request. The caller may always specify a different mode.

Partial capture

In scope

Authorise £600, capture £540, release the rest. One auth → at most one capture. The remainder is released automatically.

Out of scope

Multi-capture and incremental auth. One auth, one capture — that is the contract.

Ledger posting rules

EventLedger effectNotes
Auth (MANUAL) No posting. Auth is a hold, not revenue. Auth ≠ revenue.
Capture (MANUAL) Posts revenue entry for the captured amount. Carries pspReference + merchantReference.
Auth+Capture (IMMEDIATE) Posts revenue entry atomically. Same as capture — happens in one step.
Cancel uncaptured auth No reversal. Nothing was posted. Cancelling an uncaptured auth → no ledger entry.
Refund Posts reversal entry (reduces net revenue). Linked to original via originalReference.

Outbox / Event Log decoupling

1

A payment event (capture, refund) writes atomically to the outbox table in the same transaction.

2

The ledger service reads the outbox and applies the posting.

3

Idempotency: the outbox entry carries the pspReference. Duplicate processing is a no-op.

This avoids distributed transaction concerns while keeping the ledger eventually consistent with the payment state.

Revenue reporting

Revenue is reportable by vertical via getRevenue(window, groupBy). Refunds correctly reduce net revenue via the reversal posting. All postings carry both pspReference and merchantReference for full traceability.

Refund model

  • A Refund is a one-to-many child of Payment.
  • Each refund has its own pspReference.
  • Linked to the parent payment via originalReference (the parent/child PSP chain).
  • Partial refunds supported.
  • Cannot refund more than the captured amount.
  • Refund posts a reversal entry to the ledger.

Refund chain example

Payment  (pspReference: PSP-abc123)
  └─ Refund  (pspReference: PSP-def456,  originalReference: PSP-abc123)
  └─ Refund  (pspReference: PSP-ghi789,  originalReference: PSP-abc123)

Each refund traces back to the original payment via originalReference. The ledger service creates a reversal entry for each refund that reaches REFUNDED state.

Standing rule

Frozen contracts may not drift silently. Any deviation a builder must make is recorded here with date, the clause affected, what changed, and why. Until a row is arbitrated and the contract re-frozen, the code is the source of truth for that clause.
